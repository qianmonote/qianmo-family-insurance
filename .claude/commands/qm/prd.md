你将**智能解析**一份 PRD（产品需求文档）：自动将其拆解为若干 Future 功能模块，为每个模块生成带验收标准的 Requirements，再细化为可执行任务清单并逐项实现。本命令体现"智能 PRD 解析"——主动识别需要澄清的关键问题（如存储方案），不自行假设关键业务规则。

## 设计模式

本命令遵循 [Google ADK · 5 种 Agent Skill 设计模式](https://x.com/GoogleCloudTech/status/2033953579824758855) 中的 **Generator（模板生成器）** 模式，并组合 **Inversion** 与 **Reviewer**：

- **Generator（主）**：把 PRD 转化为四份**结构固定**的产出文档（`requirements.md` / `proposal.md` / `design.md` / `tasks.md`），每份按固定**模板（asset）**填充强制章节，并遵循质量**参考（references）**——保证每次拆解的结构与可验收性一致（见第二步）。
- **Inversion（组合）**：拆解时遇到关键未知（存储方案、第三方依赖、状态机边界、计算规则等）先**反向访谈用户**，记入"待澄清问题"；澄清前不得据假设落地（见第二步末尾与 3.5 门禁）。
- **Reviewer（组合）**：每个最小执行单元实现后，按清单对 `git diff` 评审并**按严重度分组**（阻断性 / 非阻断性），阻断项清零后才放行（见 3.3）。

PRD 路径：$1（若未提供，询问用户文档路径或让用户直接粘贴内容）

## 执行前置条件：git 仓库

开始前检查项目根目录是否为 git 仓库（`git rev-parse --is-inside-work-tree`）：

- 若不是 → 先执行 `git init` 并提交一次基线 commit（包含当前所有现有文件），再开始拆解/执行。这样单任务循环中每一步改动都能通过 `git diff` 提交评审。
- 若评审工具仍不可用，按 3.3 的规则向用户说明并询问是否跳过评审，不阻塞任务执行。

## 第〇步：判断场景

根据 PRD 内容确定本次对应的需求主题（kebab-case），检查 `Specs/<主题>/` 是否已存在：

- **不存在** → "初次新建"场景，按第一步 ~ 第三步完整执行
- **已存在** → "二次编辑修改"场景，读取该目录下现有的 `requirements.md`、`proposal.md`、`design.md`、`tasks.md`、`lessons.md`，再判断属于以下哪种情况：
  - **续接执行**：`tasks.md` 中已有"待开始/进行中"任务，且本次 PRD 内容与 `proposal.md` 记录的范围一致（未变更）→ 跳过第二步，直接进入第三步循环，从下一个未完成任务继续
  - **增量修改**：本次 PRD 相比 `proposal.md` 记录有新增/调整的需求点 → 执行以下增量流程：
    1. 对比新 PRD 与现有 `requirements.md`/`proposal.md`/`design.md`，列出新增/变更/废弃的功能模块与需求点
    2. 增量更新 `proposal.md`：在文末追加"变更记录"小节（记录本次变更内容与时间），不删除/不重写已有内容，除非用户明确要求修改某条已有结论
    3. 增量更新 `requirements.md`/`design.md`：补充/调整新增功能模块对应的验收标准、数据模型、接口、组件设计，同样以追加方式记录变更
    4. 增量更新 `tasks.md`：保留已完成任务原状；新增任务追加到列表末尾并标注依赖；若变更影响到已完成任务，将其状态改为"需复核"并说明复核原因
    5. 与用户确认增量后的文档，确认后通过 `TodoWrite` 同步新增/需复核任务，进入第三步循环（仅处理新增与"需复核"任务）

## 第一步：读取上下文（仅"初次新建"场景执行）

1. 读取 PRD 文档（$1）；若 `$1` 未提供，询问用户文档路径或让用户直接粘贴内容
2. 读取项目根目录的 `CLAUDE.md`（由 `/qm:init` 生成），了解技术栈选型、工作流架构与前后端开发规范；若不存在，提示用户先运行 `/qm:init`，但仍可继续（以浏览到的实际目录结构为准）
3. 浏览现有目录结构（前端、后端、共享包、`.claude/agents/`、`cloud/` 等），了解：
   - 当前技术栈（框架、ORM、包管理器，与 `CLAUDE.md` 记录是否一致）
   - 已有的实体模型（Drizzle schema）、API 路由、页面/组件，避免重复实现
   - 现有 `Specs/` 目录下是否已有相关提案，避免与历史拆解冲突

## 第二步：智能拆解（Generator · 模板填充，仅"初次新建"场景执行）

> **Generator 模式**：本步把 PRD 灌入四份固定模板。下面 2.1~2.6 定义每份文档的强制章节（模板）与质量规则（参考），2.7 负责按模板落盘——任何一份产出缺少强制章节都视为不合格。

### 2.1 拆解为 Future 功能模块

将 PRD 自动拆解为若干**独立的 Future 功能模块**（典型为 6 个左右，按 PRD 实际内容调整数量），每个模块是一个可独立交付、相对内聚的功能域。拆解原则：

- 按业务能力边界切分，模块之间依赖清晰、职责单一
- **大模块继续细分为子模块**：例如一个"视频模块"可细分为"播放基建"（点播/直播/转码/CDN）与"AI 能力"（字幕、内容理解、推荐）等子模块；任何聚合多个子流程的模块都应这样下钻一层
- 为每个模块/子模块标注：模块名、目标、关键能力、与其他模块的依赖

### 2.2 需求规格与验收标准（Requirements）

为每个 Future 功能模块生成**带验收标准的 Requirements**，这是后续设计与验收的依据，每个模块包含：

- 模块目标与用户故事
- 功能点列表
- **验收标准（Acceptance Criteria）**：以可验证的条目描述（Given/When/Then 或清单式），后续 QA Sub-Agent 与自动化测试据此验证
- 关键非功能需求（性能、安全、合规）如有则列出

### 2.3 需求提案（Proposal）

基于 PRD 与第一步的项目现状分析，整理一份"需求提案"，内容包括：

- 需求背景与目标（提炼自 PRD）
- Future 功能模块清单及其优先级
- 与现有系统的差异点 / 复用点（基于已浏览的目录结构）
- 范围边界（本次包含/不包含的内容）
- **待澄清问题列表（Inversion 触发点）**：需求切分时**主动反向访谈用户**确认关键问题——尤其是存储方案（用什么数据库/对象存储/缓存）、第三方依赖、状态机边界、计算规则等，不得自行假设，列入此处待用户确认

### 2.4 技术方案设计（Design）

基于提案与项目现有代码，整理一份"技术方案设计"，作为后续任务拆解与实现的依据，内容包括：

- **数据模型设计**：新增/修改的实体、字段（含类型、是否必填、默认值）、表关系、索引；以 Drizzle schema 片段或表格描述
- **API 设计**：每个新增/修改接口的 method + path + 入参/出参（zod schema 草案）+ 鉴权要求 + 错误码
- **前端页面/组件设计**：页面路由、关键组件拆分、依赖的接口、主要状态与交互流程；若 PRD 中提供了 Figma/Stitch 设计稿链接，为对应页面补充"设计稿"字段记录该链接，供执行阶段的"设计稿还原"流程使用
- **关键业务逻辑**：计算规则、状态机/流程图（文字描述或 Mermaid）
- **存储 & 第三方集成**：依据 2.3 中已澄清的存储方案落地（数据库/对象存储/缓存选型），以及涉及的外部系统及交互方式
- **云资源**：本模块需要的云资源（如 AWS 服务），记录到 `cloud/`，供 `/qm:ai` 阶段通过 MCP 自动配置
- **风险与权衡**：技术选型/实现方式上的取舍说明（如有）

### 2.5 任务拆解（Tasks）

依据 2.4 的技术方案，将每个 Future 功能模块拆解为可执行的开发任务，每个任务粒度控制在"半天内可完成"，并与具体设计项对应：

- **数据模型变更**：新增/修改的实体、字段、表关系（对应 Drizzle schema 改动）
- **后端接口**：每个接口一个任务，注明 method + path + 入参/出参（zod schema）
- **前端页面/组件**：每个页面或可复用组件一个任务，注明依赖的接口
- **共享类型/校验规则**：前后端共用的 schema 改动
- **其他**：权限控制、定时任务、第三方集成、云资源配置等

每项任务包含：序号、所属 Future 模块、模块类型（前端/后端/共享/云）、任务描述、依赖关系、对应验收标准编号。

### 2.6 任务拆分为子任务（可选）

若某项任务粒度仍超过"半天可完成"，将其拆分为子任务：

- 子任务编号采用 `<父任务序号>.<子序号>`（如 `3.1`、`3.2`）
- 子任务同样注明模块、描述、依赖关系（可依赖同父任务下的其他子任务）
- 父任务本身不再单独执行，仅作为分组标题；执行与状态跟踪以子任务为最小单元

### 2.7 写入 Specs 目录

在项目根目录创建（若不存在）`Specs/` 目录，按本次需求建立一个子目录（kebab-case 命名，体现需求主题），写入：

- `requirements.md`：2.1 的 Future 功能模块 + 2.2 的验收标准
- `proposal.md`：2.3 的需求提案
- `design.md`：2.4 的技术方案设计
- `tasks.md`：2.5/2.6 的任务清单（Markdown 表格），按 Future 模块分组，标注每个最小执行单元的状态（待开始/进行中/已完成/阻塞/需复核）；父任务状态为其所有子任务状态的汇总
- `design-assets/`：（可选，存在设计稿还原页面时创建）存放设计稿截图/导出图、页面截图、还原度对比报告

同时通过 `TodoWrite` 创建对应的 todo 列表，与 `tasks.md` 中的任务一一对应。

与用户确认 `requirements.md`、`proposal.md`、`design.md`、`tasks.md` 内容后再开始执行。如果 PRD 中有歧义或缺失信息（尤其是 2.3 中列出的存储方案/关键业务规则），先向用户提问澄清并记录在 `proposal.md` 的"待澄清问题"中，不要自行假设。

> **门禁（Inversion Gate）**：只要 2.3 的"待澄清问题"中存在会影响数据模型 / 接口 / 关键业务规则的未决项，**不得进入第三步执行**——先反向访谈用户取得答案、回填 `design.md`，再开工。

## 第三步：逐项执行（单任务循环）

按依赖顺序逐个**最小执行单元**执行（先共享类型/数据模型 → 再后端接口 → 最后前端页面/组件）。最小执行单元指 `tasks.md` 中没有子任务的任务，或某个子任务。每个最小执行单元都走完下面 3.1~3.6 的完整循环后，才进入下一个，直到 `tasks.md` 中所有任务/子任务状态均为"已完成"或"阻塞"。

> 多 Agent 协同与并行执行能力由 `/qm:ai` 提供。本命令的第三步是单任务串行循环；若需要按角色（前端/后端/QA）并行执行任务并自动加载对应 Skills，请改用 `/qm:ai`。

### 3.1 实现

- 参照 `design.md` 中对应的设计项实现当前最小执行单元，遵循 `CLAUDE.md` 中的响应格式、校验、鉴权、目录结构、表单/API 调用规则
- **设计稿还原**：若当前最小执行单元是某个页面/组件，且 `design.md` 中标注了"设计稿"（Figma/Stitch 链接），按以下流程执行：
  1. 通过 `stitch` MCP 服务读取该页面的设计稿（截图/结构化描述），作为实现的视觉与结构参照
  2. 按设计稿还原页面布局、文案、关键交互元素，同时遵循前端开发规则
  3. 若 `stitch` MCP 不可用：向用户说明并询问是否使用现有的 `ui_to_artifact` / `ui_diff_check`（zai-mcp-server）等工具替代，或跳过还原校验仅按 PRD 文字描述实现，并在 3.4 的 lessons 中记录原因
- 若涉及数据库 schema 变更，生成迁移文件但不自动执行迁移（除非用户要求）
- 关键逻辑实现后简要说明计算逻辑，方便用户核对业务正确性
- **实现与设计不一致**：若实现过程中发现 `design.md` 需要调整，先更新 `design.md` 对应章节，再继续实现，并在 3.4 的 lessons 中记录调整原因
- **执行中发现任务粒度过大**：若发现当前任务实际工作量明显超过"半天"，暂停实现，按 2.6 规则拆分为子任务并写入 `tasks.md`，与用户简要说明拆分结果后，从拆分出的第一个子任务重新进入 3.1

### 3.2 自动化测试

- 为本次改动补充/更新自动化测试，并**以 `requirements.md` 中对应的验收标准为断言依据**：
  - 测试框架统一使用 [Vitest](https://vitest.dev/)；若目标 app/package 尚未配置，按需添加 `vitest` devDependency 及最小配置
  - 测试文件与被测文件同目录，命名为 `<file>.test.ts`
  - 后端：优先覆盖 `src/services/` 中的纯逻辑与 `src/routes/` 关键分支（鉴权失败、参数校验失败、业务错误码）；数据库依赖较重的路径可视情况 mock
  - 前端：优先覆盖 `src/lib/`、`src/hooks/` 中的纯逻辑与格式化函数；组件级测试按价值取舍
- 若该最小执行单元确实不适合写测试（如纯静态文案/导航调整），在 3.4 的 lessons 中说明原因后跳过
- **页面还原度验证**：若 3.1 中执行了"设计稿还原"，额外补充：
  - **视觉还原度**：用 Playwright（或项目已有 e2e 工具）对实现页面截图，与 `stitch` 设计稿截图通过 `ui_diff_check`（或同类工具）比对，记录明显差异
  - **结构/文案还原度**：核对关键 DOM 结构、文案、布局元素与设计稿是否一致，可通过 Playwright/Testing Library 断言
  - 若无可交互真实环境，还原度验证可降级为"结构/文案"层面，并在 3.4 的 lessons 中说明原因
  - 验证产出存放到 `Specs/<本次需求目录>/design-assets/`

### 3.3 代码评审（Reviewer 模式）

- **Reviewer 模式**：按固定清单逐项评审、并**按严重度分组**输出结论（阻断性 / 非阻断性），而非泛泛而谈
- 对本次改动（`git diff`，含新增测试）调用代码评审工具进行评审，关注正确性、与 `CLAUDE.md` 规范一致性、安全性，以及**是否存在"未消费的必填字段"等逻辑漏洞**
- 若评审发现阻断性问题：修复后重新评审，直至无阻断性问题（**门禁：阻断项未清零不得标记完成**）；非阻断性建议记录到 3.4 的 lessons 中
- 若评审工具不可用，向用户说明并询问是否跳过本次评审
- 更深入的三种 Code Review 模式（基础/对抗/Git 差异）由 `/qm:ai` 提供

### 3.4 标记完成 + 总结 lessons

- 用 `TodoWrite` 标记当前最小执行单元为已完成，同步更新 `tasks.md` 中对应任务/子任务状态为"已完成"；若所属父任务的所有子任务均已完成，将父任务状态也汇总更新为"已完成"
- 将本次的经验教训（评审非阻断建议、踩坑点、关键决策依据、可复用模式、设计稿还原差异等）追加写入 `Specs/<本次需求目录>/lessons.md`

### 3.5 评估是否继续（Inversion Gate）

- 检查 `proposal.md` 中是否存在阻塞下一个最小执行单元的"待澄清问题"
- 若存在阻塞：停止循环，**反向访谈用户**汇报当前进度与阻塞点，等待澄清后再继续
- 若不存在阻塞且 `tasks.md` 中仍有未完成的任务/子任务：继续 3.6

### 3.6 上下文管理，进入下一项

- 当前最小执行单元的状态、产出、经验教训均已持久化在 `Specs/<本次需求目录>/` 中，上下文可安全压缩/清空
- 提示用户当前项已完成并已记录，可执行 `/compact` 压缩上下文，或 `/clear` 后重新运行 `/qm:prd $1` 续接
- 续接时：读取 `tasks.md`，跳过已完成任务/子任务，从下一个"待开始"的最小执行单元进入 3.1，循环执行

## 完成后

- 汇总本次完成的任务清单（已完成/未完成），并确保 `tasks.md` 状态与实际一致
- 列出涉及的文件改动
- 汇总 `lessons.md` 中本轮新增的经验教训
- 提示用户：
  - 数据库迁移是否需要执行
  - 是否需要补充测试 / 验收标准是否全部覆盖
  - 未完成或被澄清问题阻塞的任务（对应 `proposal.md` 中的"待澄清问题"）
  - 如需多 Agent 并行 + 三种 Code Review + 云资源自动配置，使用 `/qm:ai` 驱动
